## 1. 一句话定位

> **Flowshot 是一个 Node CLI 工具。开发者在自己的 Web 项目里安装它（dev dependency），用一份配置 + 一个 seed 钩子声明「要把哪些页面、按什么顺序、连成什么流程图」，运行两条命令即可产出一份单 HTML 评审稿。**

### 1.1 关键词

| 词 | 含义 |
|----|------|
| CLI | 不是 Web 服务、不是库、不是 GUI；命令行入口 `flowshot capture` / `flowshot build` |
| dev dependency | `npm i -D flowshot`，跟着用户项目走，不是全局工具 |
| 配置 + seed 钩子 | Flowshot 自己不懂业务，所有「怎么造数据、怎么登录」由用户在 `flowshot/seed.mjs` 里写 |
| 两条命令 | `capture`（截图）→ `build`（打包），中间产物（截图 + manifest）可检查可重跑 |
| 单 HTML 评审稿 | 唯一交付物形态；不做 PDF、不做在线分享 |

### 1.2 非定位（边界硬约束）

- ❌ 不是 Storybook 替代品（Storybook 是组件库，Flowshot 是页面流）
- ❌ 不是 Figma 替代品（Figma 是设计稿，Flowshot 是真截图）
- ❌ 不是 E2E 测试工具（用 Playwright 但不做断言）
- ❌ 不做协作/评论/在线签批
- ❌ 不做 PDF / Word / 在线分享链接

### 1.3 目标用户（3B 佛系开源）

- **主要用户**：作者自己 —— 把自有 Web 项目交付给客户评审时使用
- **次要用户**：在 GitHub 上发现并愿意尝试的其他开发者；提 issue 看心情回应；不做主动推广

## 2. 用户旅程

### 2.1 首次使用（V1 无 `init` 命令，README 给可粘贴模板）

```bash
# 1. 在自己的 Web 项目里
cd my-website
npm i -D flowshot

# 2. 手动建 flowshot/ 目录，复制 README 模板
mkdir flowshot
# 编辑 flowshot/flowshot.config.ts —— 声明节点、边、dev server
# 编辑 flowshot/seed.mjs       —— 写自己项目的造数据逻辑

# 3. 跑起来
npx flowshot capture     # 启动 dev server → 逐节点截图 → 存 flowshot/.cache/
npx flowshot build       # 读 cache + config → 产出 flowshot/dist/index.html

# 4. 把 dist/index.html 发给客户
```

### 2.2 日常迭代

| 场景 | 命令 |
|------|------|
| 改了某个页面 UI | `flowshot capture --only N3,N4` → `flowshot build` |
| 加了一个新节点 | 编辑 config → `flowshot capture` → `flowshot build` |
| 调流程图布局（节点位置） | 改 config 里的 `x/y` → `flowshot build`（不重截图） |

### 2.3 V1 故意不做的体验

| 砍掉项 | 理由 |
|--------|------|
| `flowshot dev`（拖节点的可视化编辑器） | 手填坐标够用；做了容易返工 |
| `flowshot init`（脚手架命令） | README 给完整可粘贴示例，省一周工作量 |
| 自动启浏览器看结果 | `build` 完打印 `file://...` 路径，用户自己双击 |

### 2.4 关键约束

- **Flowshot 不假设 dev server 怎么启动**：用户在 config 里告诉它 `command` 和 `url`，Flowshot 负责拉起 + 等就绪 + 跑完 kill
- **Flowshot 不假设怎么造数据**：把 Playwright 的 `page` 和 `context` 对象交给用户的 `seed.mjs`，用户随便用 Playwright API

## 3. 核心命令与配置契约

### 3.1 CLI 命令（V1 共 2 个）

| 命令 | 作用 | 关键参数 |
|------|------|---------|
| `flowshot capture` | 启动 dev server → 对每个节点跑 seed → 截图 → 存 cache | `--only <ids>` 只截指定节点；`--config <path>` 自定义配置路径 |
| `flowshot build` | 读 cache + config → 生成单 HTML | `--out <path>` 输出路径（默认 `flowshot/dist/index.html`） |

**退出码**：成功 0，配置错误 1，截图失败 2，构建失败 3（方便 CI 集成）。

### 3.2 `flowshot.config.ts` 配置契约

V1 同时支持 `.ts` 和 `.js`，运行时由 `tsx` 加载（Vitest/Nuxt 同款方案，无需用户配 tsconfig）。

```ts
// flowshot/flowshot.config.ts
import { defineConfig } from 'flowshot'

export default defineConfig({
  // 项目元信息（写入版本徽章）
  project: {
    name: 'CrediOS Sales Workspace',
    scope: 'Sales Officer · NTC + ETC',
  },

  // dev server 启动（Flowshot 负责拉起 + 等就绪 + kill）
  devServer: {
    command: 'npm run dev',
    url: 'http://localhost:3000',
    readyTimeout: 30_000,
  },

  // 截图参数
  capture: {
    viewport: { width: 1280, height: 800 },
    format: 'webp',
    quality: 85,
  },

  // 节点清单（id 必须唯一，x/y 为画布坐标）
  nodes: [
    {
      id: 'W1',
      title: 'Data Entry Workbench',
      route: '/sales-workspace',
      desc: '待办列表 + New Application CTA',
      x: 100, y: 200,
    },
    {
      id: 'A1',
      title: 'Identity Scope',
      route: '/applications/new',
      seed: 'identity-scope',     // 对应 seed.mjs 里导出的函数名
      x: 400, y: 200,
    },
  ],

  // 边
  edges: [
    { from: 'W1', to: 'A1', label: 'New Application' },
    { from: 'A1', to: 'N1', label: 'NTC' },
  ],
})
```

### 3.3 `seed.mjs` 钩子契约

```js
// flowshot/seed.mjs
// 每个 key = 一个 seed 名字；Flowshot 在跑对应节点前调用
// page = Playwright Page；context = BrowserContext
export default {
  'identity-scope': async ({ page, context }) => {
    await context.addInitScript(() => {
      localStorage.setItem('application_draft:flow-IDENTITY',
        JSON.stringify({ id: 'flow-IDENTITY', stage: 'IDENTITY_SCOPE' }))
    })
  },

  'ntc-de2-entry': async ({ page, context }) => {
    await context.addInitScript(() => {
      localStorage.setItem('application_draft:flow-NTC-DE2', '...')
      localStorage.setItem('ntc-de2-entry:flow-NTC-DE2', '1')  // 解锁键
    })
  },
}
```

**关键设计点**：

1. **Flowshot 不内置任何 seed adapter** —— localStorage / cookie / API helper 全砍。用户用 Playwright 原生 API 自由发挥
2. **节点 → seed 显式映射**：节点上写 `seed: 'xxx'` 指向 seed.mjs 的 export key；没声明 seed 的节点直接 `page.goto(route)` 截图
3. **route 占位符不做**（V1）：节点 `route` 写死字符串。要动态拼路径就在 seed 里用 `page.goto()`

### 3.5 技术选型

Flowshot 的技术栈由 **「自包含单 HTML 交付 + 流程图画布」** 两条硬约束推导得出。完整验证记录见 `docs/01.Spike S5 报告.md`（结论：PASS，4 节点 demo 单文件 337KB / gzip 107KB，`file://` 协议零控制台错误）。

| 层 | 选型 | 理由 |
|----|------|------|
| **流程图画布** | [`@xyflow/react`](https://reactflow.dev) (React Flow, MIT) | 节点 + 边 + 自动路由 + label 自动定位 + MiniMap + Controls + 平移缩放全套开箱即用；行业事实标准；社区活跃 |
| **UI 框架** | React 18 | React Flow 是 React 原生组件；与 Vite 生态无缝 |
| **构建工具** | Vite 5 + `@vitejs/plugin-react` | 构建快、配置简、SSR 不需要 |
| **单文件打包** | [`vite-plugin-singlefile`](https://github.com/richardtallent/vite-plugin-singlefile) | JS/CSS 全部 inline 进单个 HTML，无外部资源引用 |
| **Lightbox** | 原生 `<dialog>` 元素 | 标准 API，零依赖，ESC 自动关闭 |
| **截图自动化** | Playwright | 行业标准；支持 `addInitScript` 在浏览器上下文注入 seed；用户已熟悉 |
| **图片压缩** | `sharp` | Node 原生；WebP q=85 比 PNG 省 60-70%；批处理快 |
| **CLI 解析** | `commander` | 体积小、API 稳定 |
| **TS 配置加载** | `tsx` | Vitest/Nuxt 同款方案；运行时编译 `.ts` config 无需用户配 tsconfig |
| **包构建** | `tsup` | esbuild 驱动，零配置打 ESM + CJS + dts |

### 3.6 关键技术决策

1. **画布层为何不自己手写 SVG / 不用 D3 / 不用 panzoom**
   - 自己写 SVG：18+ 节点 + 分叉拓扑的边路由、label 定位、平移缩放是巨大工程，Spike 早期方案（v0.5 用 panzoom + 手写 SVG）已被否决
   - D3：偏数据可视化，节点 + 边的交互式编辑生态不如 React Flow
   - React Flow：直接覆盖所有需求，节省 90% 自研工作量

2. **构建产物为何必须单 HTML**
   - 评审稿要走邮件/网盘投递，客户解压双击就开，**不能依赖 HTTP 服务器**（`file://` 协议下 `fetch()` 会触发 CORS 失败）
   - `vite-plugin-singlefile` 把所有 JS/CSS inline；截图用 `<img src="data:image/webp;base64,...">` 内联；manifest 用 `<script type="application/json">` 内联

3. **节点拖拽策略**：开发时 `nodesDraggable={true}` 让用户调位置，交付版 `nodesDraggable={false}` 防客户误操作

4. **`<Handle>` 必须显式声明**（Spike S5 踩坑）：自定义节点没 `<Handle type="source"/target">` 时，边渲染数 = 0。修复方案：节点内放隐藏 Handle (`opacity: 0`)

5. **图片体积**：18 张真实截图 WebP（每张约 200KB）内联后预估单文件 4–5MB，邮件附件可传

6. **截图稳定性靠工程约定，不靠换工具**
   - Playwright 已是当下最稳的浏览器自动化方案（多浏览器 + `addInitScript` + 一流 auto-wait + Trace Viewer），换工具收益微乎其微
   - 真正的 flaky 来源是「SPA hydration / 动画 / 字体 / 懒加载 / 时间戳随机数 / 用户 seed 时序」，与工具无关
   - `flowshot capture` **内置**以下默认行为，把不稳定性降到最低：
     - 注入 CSS `*{transition:none!important;animation:none!important}` 禁用所有动画/过渡
     - 等待 `document.fonts.ready` + `page.waitForLoadState('networkidle')`
     - 节点 config 支持可选 `readySelector: '[data-flowshot-ready]'`，要求用户在 SPA「真正就绪」时渲染该 DOM 信号
     - 失败自动重试 1 次（截图前清空 cookie/storage，重新跑 seed）
     - 节点之间使用独立 `BrowserContext`，天然隔离 cookie/storage 污染
   - README 提供「编写稳定 seed.mjs 的最佳实践」清单（如：先 `addInitScript` 再 `goto`、避免依赖 `setTimeout`、用 `data-flowshot-ready` 信号而非 `waitForTimeout`）

### 4.1 关键决策

| # | 决策 | 理由 |
|---|------|------|
| 1 | `build/renderer/` 是 Vite 子工程，不是用户可见产物 | 内部用编程式 `vite build`；renderer 通过 `<script type="application/json">` 注入用户数据 |
| 2 | 现有 `src/main.jsx`（Spike S5 demo）改写为 TS 搬进 `src/build/renderer/` | 4 节点硬编码改为从注入 manifest 读取 |
| 3 | `examples/demo-site/` 双用途 | 既是 E2E 测试目标，也是 README 「跑得起来」的示例 |
| 4 | Playwright 走 `peerDependency` | 避免重复装浏览器；用户 `npm i -D playwright flowshot` |
| 5 | renderer 依赖（React/Vite/React Flow）走普通 `dependency` | 用户不该感知内部实现 |

### 4.2 与现有仓库状态的差异

| 现状 | V1 后 |
|------|------|
| 根目录直接是 Vite Spike 工程（`src/main.jsx`、`vite.config.js`） | 根目录是 npm 包工程（`src/cli.ts` 入口） |
| `src/main.jsx` 硬编码 4 节点 | 改写为 `src/build/renderer/main.tsx`，从 manifest 读数据 |
| `verify.mjs` 是 Spike 验证脚本 | 移到 `tests/spike-verify.mjs` 或删除（已被 examples/demo-site 覆盖） |
| `package.json` 名 `uiflow-spike-s5` | 改名 `flowshot` 或 scoped `@xazaj/flowshot`（npm 名字策略见 §5.2） |

## 5. 版本、发布与维护策略

### 5.1 版本号语义

| 版本段 | 含义 | 触发场景 |
|--------|------|---------|
| **0.x.y** | V1 期间统一停在 `0.x` 段 | 表明「能用但 API 不保证稳定」 |
| **x ↑** | break change | 砍 deprecated 字段；改 `defineConfig` 形状 |
| **y ↑** | 新功能或 bugfix | 加 CLI flag、加 dev 命令、修 bug |

**V1 首版发 `0.1.0`**；何时发 `1.0.0` —— 「自己连续 3 个真实项目都用得舒服」时再发，不设时间表。

### 5.2 npm 包名策略（待定）

> ⚠️ `flowshot` npm 包名已被另一项目占用（创建于 2026-03-21，月下载 154，GitHub 仓库 404 疑似弃坑）。

候选方案（**待用户决策**，不影响 V1 开发；发布前再定）：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **A. scoped 包 `@xazaj/flowshot`** | 不改项目/仓库名 | npm 搜索仍有混淆 |
| **B. 暂不发 npm** | V1 期间用 `npm i github:xazaj/Flowshot` 自用足够 | 别人发现成本高 |
| **C. 改 Flowshot 项目名** | 干净 | 改名工作量 |

## 7. 风险与未决问题
